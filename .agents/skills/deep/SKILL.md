# Deep Read to Notes

## 交付契约

- 把“帮我读一下”及同义请求视为完整交付授权：读取材料、追踪关键来源、形成判断、直接讲解、写入站内笔记、验证、提交并推送。
- 优先服从用户对本次任务的明确边界。用户若说“只总结”“先别写笔记”“不要提交”或“不要推送”，只执行授权范围。
- 不因普通不确定性中断流程。先查上下文和一手资料，再使用可用检索工具补足证据。
- 不把“抓到了摘要”当作“读完全文”，不把作者陈述当作独立验证，不把相关性写成因果性。
- 遇到付费墙、登录限制、删除内容、缺失附件或推送权限问题时，继续完成可安全完成的部分，明确证据边界和未完成项，禁止猜测性补全。

## 阶段 0：建立任务边界

1. 完整读取仓库根目录 `AGENTS.md`、`.agent/codex-experience-profile.md`、`Progress.md`、`.agent/skills/notes-authoring/SKILL.md`、`notes/NOTE_TEMPLATE.md`；需要新增索引时读取 `_data/notes.yml`。
2. 检查 Git 分支与工作区状态，区分本任务文件、用户已有改动和其他并行任务改动。禁止回滚、覆盖或顺手提交无关内容。
3. 建立分阶段计划，并在 `Progress.md` 记录任务目标、材料边界、关键判断、变更和验证结果。
4. 识别材料类型、作者、规范标题、发布日期、版本和 canonical URL。链接不明确时先定位原始出处。
5. 把下载材料、转录、抓取缓存和实验输出放在仓库外的临时目录；仓库只保留最终笔记及确有阅读价值的本地图片资产。

## 阶段 1：构建高价值来源图

先完整获取主材料，再沿着会影响理解或结论的链接扩展证据图。遵循仓库 `AGENTS.md` 的检索路由与浏览器安全规则；默认不控制用户前台 Chrome，工具降级时记录原因。

### 读取主材料

- 论文：读取正文、脚注、图表、附录和补充材料；优先核对 arXiv/出版社版本、TeX 或结构化全文、官方代码、数据集、模型卡和项目页。只读 abstract 不算完成。
- 推文或 thread：读取完整串文、引用帖、作者补充回复和所有实质性外链；搜索结果摘要或单张截图不算完整原文。
- 博客、技术报告或访谈：读取完整正文或转录，核对作者身份、发布日期、修订记录、演示材料、代码和原始数据。
- 二手解读：只用于定位争议、反例或行业背景；关键事实尽量回到一手来源。

### 追踪内链与参考文献

优先打开以下高价值来源：

- 主材料理解所依赖的方法、公式、数据集、benchmark 或前置工作；
- 支撑 headline claim 的实验、表格、代码、模型配置或原始数据；
- 官方仓库、issue、作者澄清、勘误、后续版本和可复现材料；
- 会显著支持、削弱或改变核心判断的批评、反例与独立复现。

不要机械遍历所有 bibliography。满足以下条件后停止扩展：主材料已完整读取；关键结论均有来源或被标为未核验；新增链接不再实质改变问题定义、机制、证据强度、限制或独立 insight。

## 阶段 2：建立证据账本并深度分析

在工作区外或会话上下文中维护私有证据账本，为每个关键判断记录“主张、来源位置、证据状态、限制”。至少区分：

- `已核验事实`：可由原文、代码、数据或独立计算直接支持；
- `发布方报告`：来自作者或机构，但本轮未独立复现；
- `分析推断`：由多项证据推导，必须说明推理链；
- `建议`：面向研究或工程实践的行动判断。

完成以下分析，而不是只改写摘要：

1. 说明它试图解决的真实问题，以及旧方法为什么不够。
2. 把核心机制讲成输入、处理、输出、训练信号或控制流，并解释首次出现的术语和公式符号。
3. 区分真正的新贡献、已知组件的工程组合和营销性命名。
4. 核对关键图表、样本量、基线、公平性、指标口径、消融、统计不确定性和失败案例。
5. 对重要比例、参数量、成本或增益做可行的独立复算；写清假设，不制造伪精度。
6. 在结论依赖实现细节时检查官方代码、配置、数据处理或评测脚本，比较“论文怎么说”和“代码怎么做”。
7. 评估复现条件、外部有效性、缺失证据、替代解释，以及什么新证据会推翻当前判断。
8. 形成至少一条超越原文复述的独立 insight，并解释为什么重要、适用边界和工程含义。

## 阶段 3：形成简洁但全面的解释

采用“结论先行”的中文表达，通常按以下顺序组织：

1. 一句话核心判断；
2. 问题背景与价值；
3. 核心机制；
4. 最有说服力的证据；
5. 关键限制与反例；
6. 独立 insight 与实践含义。

删除不改变理解的枝节，保留会改变结论的数字、假设和证据边界。不要用术语堆砌代替解释，也不要为了简洁省略关键限制。最终回复必须直接向用户讲清楚核心内容，不能只给一个文件链接。

## 阶段 4：沉淀站内笔记

1. 严格遵循 `.agent/skills/notes-authoring/SKILL.md` 和 `notes/NOTE_TEMPLATE.md`。
2. 先搜索是否已有同主题笔记；能原地深化时不要创建重复版本。确需新增时，论文类优先放 `notes/paper-reviews/`，技术帖、访谈或跨材料分析优先放 `notes/tech-analysis/`。
3. 让文章开头直接进入主题判断、问题、机制和结论；把公开来源、版本信息和证据边界统一放在文末 evidence appendix。
4. 更新 `_data/notes.yml`；本地化确有解释价值的图片，提供准确 `alt`；只在存在公式时加载 MathJax。
5. 不在公开笔记中暴露抓取命令、工具名、生成过程、临时文件、本地绝对路径或提交过程。
6. 更新 `Progress.md`，记录研究覆盖、关键判断、文件变更和真实验证结果。

## 阶段 5：验证内容与页面

至少完成以下检查：

```bash
ruby "scripts/validate_notes_index.rb"
git diff --check
```

- 对本次笔记定向扫描生成痕迹、本地路径、替换字符、空图片 `alt` 和明显占位文本。
- 检查唯一 `<main>`、标题层级、重复 ID、目录断锚、本地图片、MathJax 条件加载和文末 evidence appendix。
- 新增笔记、索引、资产或结构变更时，使用隔离输出目录完成 Jekyll build，避免共享 `_site` 造成竞争或污染。
- 使用专用 headless/in-app 浏览器在桌面与手机视口渲染新增或大改页面；检查整页横向溢出、宽表局部滚动、坏图、公式、锚点、控制台错误和失败请求。不要占用用户前台 Chrome。
- 最后重读笔记，逐项核对数字、来源归属、事实/推断边界和标题是否夸大。
- 只把已确认与本次变更无关的既有 warning 作为已知边界记录；新错误必须修复后重新验证。

## 阶段 6：隔离提交并推送

1. 查看最终工作区 diff，列出本任务拥有的文件和共享文件中的本任务 hunk。
2. 只暂存本任务文件或 hunk；当 `Progress.md`、`AGENTS.md`、`_data/notes.yml` 等文件含有并行改动时，使用选择性暂存，禁止把无关改动带入提交。
3. 检查 staged diff、文件清单和提交信息，确保公开笔记没有生成痕迹或敏感信息。
4. 使用聚焦的提交信息（通常为 `docs(notes): ...`）提交，并推送当前分支。禁止 force push。
5. 验证本地提交、远端分支和 upstream 同步状态。若推送因权限、保护规则或无法安全解决的远端分叉失败，保留已验证的本地提交并报告精确阻塞点。

## 完成门槛

只有同时满足以下条件，才把默认完整流程报告为完成：

- 主材料已完整读取，或者无法访问的部分已明确标注；
- 会改变核心结论的关键内链和参考资料已核验；
- 原文事实、发布方报告、分析推断和建议没有混写；
- 已向用户直接给出简洁但全面的中文讲解；
- 笔记、索引和 `Progress.md` 已正确更新；
- 内容、结构、构建和桌面/手机渲染验证与改动风险相匹配；
- 只提交本任务改动，提交已成功推送；
- 最终报告包含材料边界、关键 insight、文件、验证结果、commit 和 push 状态。
