## 一、执行经验总结

### 1. 不先读仓库规则会导致错误工作流

问题做法：

- 直接按通用经验行动，未先读 `AGENTS.md`、`CLAUDE.md`、`Progress.md` 或项目内最新流程。
- 在已有仓库规则存在时重新发明目录、命令或文档结构。
- 忽略已有脏工作树，导致把无关改动混入当前任务判断。

正确方式：

- 每次进入仓库先读根目录 `AGENTS.md`、`Progress.md`，必要时读 `CLAUDE.md`、README、任务相关 docs。
- 先 `git status --short`，明确哪些改动属于当前任务、哪些是已有上下文。
- 对代码判断先读相关实现，再下结论；不能用测试绿灯替代代码理解。

教训：

- 对这个用户，正确性来自“当前证据链”，不是来自通用习惯。先读规则是工作的一部分，不是前置开销。

### 2. 乱用搜索工具会制造噪声和错误

问题做法：

- 对整个仓库或 `~/.codex` 做过宽 `rg`，把 minified 静态文件、缓存、历史 worktree、长指令注入一起扫进来。
- 搜索公开网页时没有优先使用项目约定的 `opencli` adapter，导致抓取不完整或来源不可复现。
- 把工具不可用、网页验证码、API 429 等情况默默吞掉。

正确方式：

- 本地文件搜索优先 `rg`，但先限定目录和文件类型；对大仓库先做 `rg --files`、`find -maxdepth`、代表性文件抽样。
- 公开网页、X 线程、arXiv、GitHub、YouTube、知乎等材料优先 `opencli`；先看 `opencli <site> --help -f yaml`，失败再降级。
- 降级必须说明原因：adapter 不存在、登录态不足、验证码、API 限流、返回信息不足。

教训：

- 搜索不是越大越好。高质量检索要先缩小证据域，保证输出可以复现、可以解释、可以被再次核验。

### 3. 未按阶段验证会把错误拖到最后

问题做法：

- 大批量改完再测试，最后才发现基础命令、路径、数据缓存或 UI 启动失败。
- 只跑单个窄测试就声称整体完成。
- 对算法/策略变更只看一个样本或一个数据集，误把偶然收益当成真实改进。

正确方式：

- 任务拆阶段：读上下文 -> 定位问题 -> 小步修改 -> 运行最小验证 -> 扩大验证 -> 更新文档。
- 每阶段结束都记录命令、结果、异常和下一步。
- 优化类任务必须做 baseline 对比，覆盖代表性数据集/场景；性能优化必须比较真实端到端耗时，而不是只测局部微基准。

教训：

- 阶段验证不是仪式。它能防止“看起来推进很多，实际上偏离目标”的情况。

### 4. 目录污染和重复文档会降低长期可维护性

问题做法：

- 每轮都新建一个版本文档或临时报告，导致仓库根目录和 notes 目录膨胀。
- 把日志、缓存、下载数据、大文件写入代码仓库。
- 对已经存在的文档不做 in-place 修订，而是复制出新版本。

正确方式：

- 仓库根目录保持简洁；只有用户明确要求或有长期价值时才新增文档。
- 研发过程优先更新 `Progress.md`；已有专题文档能承载时优先原地修改。
- 大日志、缓存、数据、模型、临时下载文件放仓库外，文档只保存结论、命令和索引。

教训：

- 文档越多不等于知识越强。对这个用户，更重要的是“少而准、可定位、可延续”。

### 5. 只写笔记不直接讲解，会造成交互断层

问题做法：

- 完成阅读或分析后只生成笔记文件，没有直接向用户解释核心机制和 insight。
- 把来源、抓取过程、生成时间放在文章开头，读者进入正文前先看到工具痕迹。
- 论文/技术帖只做摘要，没有把方法动机、核心机制、实验边界、工程启发讲清楚。

正确方式：

- 对外部材料：先直接向用户清晰讲解，再根据需要沉淀到站内笔记。
- 站内笔记开头直接进入主题判断、问题背景、机制和结论；来源和核验命令放文末。
- “展示 insight”意味着解释为什么重要、哪里反直觉、工程上如何用、边界在哪里。

教训：

- 用户要的是能直接阅读和理解的分析，不是资料搬运。笔记是沉淀，口头报告是协作界面，两者都需要。

### 6. UI 不能只“能跑”，必须展示真实状态

问题做法：

- UI 只显示“运行中”或固定文案，不显示当前时间、进度、持仓、剩余资金、盈亏、正在扫描的对象、后台 PID、日志路径等关键状态。
- 前端和后台启动流程割裂，导致用户不知道启动的到底是什么。
- 只做装饰性页面或复杂 UI，而没有优先解决核心工作流。

正确方式：

- 工具类 UI 先保证功能：启动、停止、刷新、进度、状态、日志、错误可见。
- 页面保持聚焦和简洁；必要时一页只管理一个策略/任务，避免信息互相遮挡。
- 启动脚本要清理旧进程、处理端口占用、启动后台和前端，并给出可访问 URL 与日志位置。

教训：

- 这个用户重视操作透明度。UI 的首要价值是让运行状态可验证，不是展示视觉复杂度。

### 7. 性能优化不能牺牲语义和精度

问题做法：

- 为了提速加入“offline-only”等开关，却没有清晰定义缺数据时的失败语义。
- 用小样本微基准代替真实任务耗时，导致优化方向错误。
- 用缓存或并行化改变原有结果，却没有证明结果一致。

正确方式：

- 优化前先定义等价性约束：输入相同、输出一致、收益/指标不变或明确解释差异。
- 加速前后必须比较完整任务：加载、预热、计算、写日志、汇总。
- 缺数据时先检查本地覆盖率；该下载就下载，该失败就显式失败，不能静默编造或跳过。

教训：

- 性能优化的底线是正确性。没有等价性证明的加速不是优化，是风险。

### 8. 声称完成前必须做要求级审计

问题做法：

- 只因为主要代码已改或某个命令通过，就说任务完成。
- 没有逐条对照用户目标中的显式交付物、验证项和配置要求。
- 忽略用户要求的持久加载、文档沉淀或 Progress 记录。

正确方式：

- 完成前列出客观要求：独立文件、三类内容、`.agent`/AGENTS 引用、验证命令、Progress 更新。
- 对每项找证据：文件存在、路径正确、内容覆盖、配置可被后续会话发现、命令无错误。
- 证据不足就继续补，不把“没发现明显问题”当成完成。

教训：

- 最终答案必须经得起逐条验收。完成是证据状态，不是主观感觉。

## 二、用户偏好与理念档案

### 沟通风格

- 默认中文沟通；报告、日志、研究总结也用中文。
- 用户能接受直接、专业、技术密度高的表达，不需要铺垫和安慰。
- 回答要有具体判断、证据和下一步；不要只给泛泛建议。
- 用户频繁要求“更深入”“详细”“展示 insight”，说明浅层摘要不够。
- 当用户已授权“直接开干”“我要去睡觉”时，不要中途频繁请示。

### 工程理念

- 事实驱动：先读文件、日志、代码、测试结果，再判断。
- 小步闭环：计划、实现、测试、日志审计、复盘记录要持续迭代。
- KISS/YAGNI/DRY/SOLID 是默认工程约束，尤其反感无必要复杂化。
- 不接受隐藏失败：异常、缺数据、启动失败、日志错误都要暴露并处理。
- 任务完成必须包含可复现命令、验证结果和清晰变更记录。

### 产品与 UI 偏好

- 工具型产品应安静、聚焦、信息密度适中，优先服务真实操作流程。
- UI 不需要过度打磨视觉，但要有完整控件、清晰状态、可操作路径和错误反馈。
- 用户更关心“当前发生了什么”：时间、进度、后台状态、资源、日志、结果对比。
- 避免空泛介绍、营销式页面和只展示概念的卡片堆叠。
- 复杂功能要拆成主流程、批量流程、日志/报告视图，而不是把所有信息混成一屏。

### 研究与内容偏好

- 阅读论文、博客、X 线程时，要核验原始来源、作者、代码、数据集、实验表格和评论区关键异议。
- 用户重视机制解释：不只说“方法是什么”，还要说“为什么这样设计”“损失函数/训练信号如何工作”“边界是什么”。
- 笔记要可直接阅读：结论和机制在前，资料索引在后。
- 对外部材料的价值判断要明确：哪些是新贡献，哪些只是工程组合，哪些证据仍弱。

### 仓库与文档偏好

- `Progress.md` 是研发过程主线，应记录正在做的工作、关键判断、完成变更和验证结果。
- `AGENTS.md` 是稳定规则入口，不应塞满逐版本实验细节；长过程放独立文档并引用。
- 当前个人站仓库不再默认写 Obsidian；需要沉淀外部材料时优先放当前仓库合适位置。
- 文档修改优先 in-place，不为同一主题反复新建版本。
- 新增文档必须有明确长期价值；否则直接在对话中回答。

### 自主工作模式偏好

- 当用户说“我要去睡觉”或给出同等授权，必须进入持续自主模式。
- 自主模式下由 Codex 做产品和技术决策，维护 todo，持续修复、测试、查日志。
- 停止条件是代码和运行状态被验证正确，日志无异常；不是“时间到了”或“已经做了很多”。
- 不确定时先查当前上下文文档，再查官方/网络资料。

## 三、可复用规则清单

### 启动规则

- [ ] 先读 `AGENTS.md`、`Progress.md`，必要时读 `CLAUDE.md`、README、任务相关 docs。
- [ ] 先检查 `git status --short`，识别已有脏工作树。
- [ ] 明确本轮任务的实际交付物和验证证据。
- [ ] 若任务跨多步，维护简洁 plan，并随进展更新。

### 文件与命令规则

- [ ] shell 路径使用双引号，优先正斜杠。
- [ ] 搜索优先 `rg` / `rg --files`，先限定范围再扩大。
- [ ] 读写文件优先 `apply_patch`；不使用破坏性 git 命令。
- [ ] 大文件、日志、缓存、临时下载不要写入仓库。

### 搜索与资料规则

- [ ] 公开网页/论文/社媒/视频优先 `opencli` adapter；开放式检索再用 Grok-Search 或可用搜索工具。
- [ ] 每个关键事实尽量核验到原始来源。
- [ ] 工具失败要说明失败点和降级路径。
- [ ] 资料索引、抓取命令和证据边界放文末或 Progress，不放正文开头。

### 代码修改规则

- [ ] 先读相关实现，不凭猜测改代码。
- [ ] 修改范围最小化，遵循 KISS/YAGNI/DRY/SOLID。
- [ ] 保留用户已有改动，不回滚无关脏文件。
- [ ] 每阶段运行最小必要测试；涉及共享逻辑时扩大回归范围。
- [ ] 性能优化必须证明语义一致或清楚解释差异。

### UI 与产品规则

- [ ] 不做无功能 landing page；首屏优先真实可用体验。
- [ ] 工具 UI 必须显示真实运行状态、进度、错误、日志位置和可操作按钮。
- [ ] 视觉使用主流常规方案，先保证功能和信息架构。
- [ ] 状态文本、按钮和动态内容不得遮挡或溢出。

### 文档与沉淀规则

- [ ] 当前仓库研发过程优先更新 `Progress.md`。
- [ ] 长期规则放 `AGENTS.md`；详细经验放独立文档并由 `AGENTS.md` 引用。
- [ ] 非必要不新建文档；新建时要有长期复用价值。
- [ ] 修改已有文档尽量 in-place。

### 完成审计规则

- [ ] 逐条对照用户目标：内容、文件、配置、验证、报告是否完成。
- [ ] 用当前文件和命令输出证明完成。
- [ ] 测试覆盖范围必须匹配改动范围。
- [ ] 证据不足就继续工作，不提前宣称完成。
